## Passo 1 - Verificacao obrigatoria de segredos (executar SEMPRE, primeiro)

Procure no diff completo do push por padroes de segredo, incluindo mas nao se limitando a:
- Chaves de API (ex.: `sk-`, `AIza`, `ghp_`, `xox[baprs]-`, strings alfanumericas longas atribuidas a variaveis tipo KEY/TOKEN/SECRET)
- Connection strings com usuario/senha embutidos
- Blocos `-----BEGIN PRIVATE KEY-----` ou chaves SSH
- Tokens JWT completos
- Variaveis com VALOR atribuido em arquivos `.env*` (exceto `.env.example` com placeholders vazios)
- Credenciais hardcoded em qualquer linguagem
- A chave da API do ChatGPT/OpenAI
- A chave/token da API Sentinela interna
- Segredo de assinatura JWT, chaves de admin bootstrap/general, segredo de sessao

Se QUALQUER correspondencia for encontrada em QUALQUER commit do push (mesmo
que corrigida em commit posterior do mesmo push, pois o segredo ja existe no
historico Git): severity=critical, result=fail, IMEDIATAMENTE.

## Passo 2 - Verificacao de arquivos que NUNCA pertencem a este repositorio

Este repositorio (Artifice/Sentinela) e PUBLICO. Existe um repositorio
IRMAO PRIVADO, Artifice-Security, que e o unico lugar onde os arquivos
abaixo podem existir. Se qualquer um destes arquivos aparecer em QUALQUER
commit do push, isso e por definicao um erro de destino — falhar sem
analisar o conteudo:

- `docs/security/threat-model.md` (ou qualquer arquivo threat-model* em qualquer caminho)
- `.env`, `.env.local`, `.env.production`, `.env.test` (qualquer variante exceto `.env.example`)
- `*.pem`, `*.key`, `*.crt`, `*.p12`, chaves SSH privadas
- Configs de deploy com IPs/hosts reais de producao
- Dumps ou seeds de banco de dados com dados reais
- Logs de auditoria reais (contem IP, user ID, PII, acoes)
- Qualquer arquivo dentro de uma pasta `security/`, `secrets/` ou `internal/`
  que nao seja claramente documentacao generica sem dados reais

Isso vale mesmo que o arquivo tenha sido removido em um commit posterior do
mesmo push: se ele existiu em algum commit sendo enviado, ja esta no
historico Git que se tornara publico.

Para cada arquivo bloqueado neste passo, incluir em "message" a instrucao:
"Este arquivo pertence ao repositorio privado Artifice-Security, nao a
este repositorio publico. Mova a alteracao para la, remova-a do historico
deste push e reenvie."

Foque em vulnerabilidades de seguranca e em regressao de controles existentes
(rate limiting, CSP, autenticacao, CORS, HSTS ja implementados no projeto).

Regras:

- Se encontrar falhas altas ou criticas, result=fail.
- Responda estritamente no formato JSON valido conforme o schema fornecido.

Nota: esta revisao por IA e uma camada complementar, nao substitui varredura
determinística de segredos (ex.: gitleaks, truffleHog, detect-secrets) rodando
